# App-Adoption Integration Guide — CyberCad iPad app → CyberCadKernel product

**Purpose.** The concrete, sequenced plan for the CyberCad iPad app to **replace its own
OCCT bridge with the CyberCadKernel product** — the drop-OCCT campaign's #1 shipping blocker.
This is the *adoption* half of the migration (link the kernel product behind the app's existing
`cc_*` call-sites); the *flip* half (`cc_set_engine(1)` → native-only) is covered in §3(c)/§4/§5.

**Discipline.** READ-ONLY analysis. Every claim is tied to a specific line in the app
(`/Users/leonardoaraujo/work/cybercad`) or the kernel (this repo). No app or kernel product
code was modified. Where a fact could not be verified in app source, it is flagged explicitly.

**Companion docs.** This guide *acts on* the two prior audits:
[docs/APP-MIGRATION-READINESS.md](APP-MIGRATION-READINESS.md) (the architectural finding: the
app does not link the kernel + cannot call `cc_set_engine`), and
[openspec/DROP-OCCT-READINESS.md](../openspec/DROP-OCCT-READINESS.md) (the A/B/C op
classification + the §6 M8 measured native-only rehearsal). The measured payoff is in
[docs/BENCH-native-vs-occt.md](BENCH-native-vs-occt.md) and
[docs/BENCH-memory-native-vs-occt.md](BENCH-memory-native-vs-occt.md).

## TL;DR

- **Adoption is an app-integration task, not a geometry task.** The kernel exposes the same
  `cc_*` C ABI the app already calls (byte-compatible by design — `cc_kernel.h:7-15`), so the
  app keeps its Swift `BRepShape` wrapper and every call-site unchanged.
- **It is NOT a pure drop-in.** The app's own bridge header declares **6 functions the kernel
  product header does not** — and the app **actively calls all 6** (`cc_loft_circles`,
  `cc_loft_circle_wire`, `cc_loft_along_rails` [two-rail], `cc_loft_typed`,
  `cc_shape_solid_count`, `cc_shape_solid_at`). These are the exact deltas the app must resolve
  before the link swap compiles/links. See §3(b).
- **The IGES product decision is the one clean Class-C blocker the app hits** (7 live IGES
  sites). Three options costed in §3(e).
- **Adoption + flip delivers the large majority of the app's geometry natively today** (the M8
  rehearsal measured 53 host suites PASS-native / 0 crash / 0 silent-wrong). What still forwards
  to OCCT after adoption is the named A-envelope residual in §5 — smaller than the prior audit
  implied, because the M3 curved-edge fillet work has since landed for analytic-revolve substrates.

## 1. Current state (source-grounded)

**The app gets all exact B-rep geometry from its OWN in-app OCCT bridge, not from the kernel.**

- The app's `cc_*` facade is `CyberCad/Kernel/Bridge/KernelBridge.mm` (2416 lines). The entire
  C++ implementation is guarded by `#ifdef CYBERCAD_HAS_OCCT` — the OCCT `#include`s span
  `KernelBridge.mm:10-81` (71 OCCT headers: `gp_*`, `BRepPrimAPI_MakePrism.hxx:23`,
  `BRepFilletAPI_MakeFillet.hxx:28`, `BRepAlgoAPI_Fuse.hxx:25`, `STEPControl_*`,
  `IGESControl_Writer.hxx:48` / `IGESControl_Reader.hxx:49`, …).
- Each `cc_*` function calls OCCT inline. Verified: `cc_solid_extrude` (`KernelBridge.mm:596`)
  → `buildExtrusion` (`KernelBridge.mm:348`) → `BRepPrimAPI_MakePrism` (`KernelBridge.mm:352`)
  → `BRepCheck_Analyzer(solid).IsValid()` (`KernelBridge.mm:356`). The non-OCCT `#else` arm
  (e.g. `KernelBridge.mm:603-605`) returns `0` / empty mesh.
- OCCT is **vendored in the app**: `Vendor/occt/{include, lib-iphoneos, lib-iphonesimulator}`
  (48 `.a` per platform slice), linked via XcodeGen in `project.yml`:
  - `OTHER_LDFLAGS` (`project.yml:40`) lists **22 toolkits**: `-lTKernel -lTKMath -lTKG2d
    -lTKG3d -lTKGeomBase -lTKBRep -lTKGeomAlgo -lTKTopAlgo -lTKBO -lTKPrim -lTKBool -lTKFillet
    -lTKOffset -lTKShHealing -lTKMesh -lTKDESTEP -lTKDEIGES -lTKXSBase -lTKDE -lTKLCAF -lTKCAF
    -lTKCDF -lTKXCAF`.
  - `HEADER_SEARCH_PATHS` (`project.yml:37`) = `$(PROJECT_DIR)/Vendor/occt/include`.
  - `LIBRARY_SEARCH_PATHS` (`project.yml:38`) = `$(PROJECT_DIR)/Vendor/occt/lib-$(PLATFORM_NAME)`.
  - `CYBERCAD_HAS_OCCT` is defined in **two** places: `SWIFT_ACTIVE_COMPILATION_CONDITIONS`
    (`project.yml:35`) and `OTHER_CPLUSPLUSFLAGS -DCYBERCAD_HAS_OCCT` (`project.yml:36`). The
    block `project.yml:34-41` is annotated *"managed by scripts/enable-occt.sh"* — the app
    already has a script that toggles the OCCT link on/off.
  - Deployment target `18.0` (`project.yml:22`), iPad (`TARGETED_DEVICE_FAMILY: "2"`,
    `project.yml:29`). The Xcode project is **generated by XcodeGen from `project.yml`** — so
    adding a framework means editing the YAML + regenerating the `.pbxproj`, not clicking in Xcode.
- **The app has NO reference to the CyberCadKernel product.** No `libcybercadkernel.a`, no
  `cc_kernel.h`, no `NativeEngine`. Critically, the app **never calls `cc_set_engine` /
  `cc_active_engine`** (0 hits across the app), and neither does it call `cc_set_parallel` or
  `cc_set_gpu_tessellation` — the dual-engine selector exists *only* in the kernel product
  (`include/cybercadkernel/cc_kernel.h:289`). So "flip the engine to native tomorrow" is not
  even callable today: **adoption is a hard prerequisite for the flip.**
- The app's only non-OCCT fallback is a **pure-Swift `PreviewKernel`**
  (`CyberCad/Kernel/PreviewKernel.swift`), used only when `CYBERCAD_HAS_OCCT` is off. It builds
  preview meshes (extrude/revolve/sweep/loft) and explicitly does **not** provide B-rep
  fillet/boolean/shell/offset (`PreviewKernel.swift:6` — *"OCCT replaces this … for exact B-rep"*).

## 2. Target state

The app **links the CyberCadKernel product** (built by `scripts/build-xcframework.sh`) in place
of its inline OCCT bridge:

1. The app's Swift layer (`BRepShape.swift`, `Workspace.swift`, UI) is **unchanged** — it keeps
   calling the same `cc_*` names.
2. Those `cc_*` calls resolve into the kernel product's ABI (`include/cybercadkernel/cc_kernel.h`)
   instead of `KernelBridge.mm`'s inline OCCT.
3. Inside the kernel, the **NativeEngine** serves the calls; the kernel's *own* embedded OCCT
   adapter (`src/engine/occt/**`) is the transitional fallback/oracle, reached only for
   out-of-envelope input via the mandatory self-verify fall-through
   (`native_engine.cpp:575-581`).
4. During the transition the app calls `cc_set_engine(0)` (OCCT default) or `cc_set_engine(1)`
   (native) to A/B behind the same calls. At the terminal unlink the kernel is rebuilt OCCT-free
   and the app drops `Vendor/occt` entirely.

The ABI compatibility is a **contractual, tested** property of the kernel, not an accident:
`cc_kernel.h:7-15` states it mirrors `KernelBridgeAPI.h` byte-for-byte, and `tests/test_abi.cpp`
static_asserts the POD struct sizes/offsets against the reference header. The POD types the app
depends on — `CCMesh`, `CCShapeId`, `CCProfileSeg`, `CCMassProps`, `CCEdgePolyline`, `CCFaceMesh`
— are identical in both headers (verified by direct comparison).

## 3. Integration steps — sequenced and concrete

### (a) Build + embed the kernel xcframework

- **Which script / artifact.** `scripts/build-xcframework.sh` compiles every `src/**/*.cpp`
  with `-DCYBERCAD_HAS_OCCT -std=c++20 -Os -fvisibility=hidden` for **two slices** — device
  `arm64-apple-ios13.0` (OS64) and simulator `arm64-apple-ios13.0-simulator` (SIMULATORARM64) —
  archives each to `libcybercadkernel-<slice>.a`, then runs `xcodebuild -create-xcframework` to
  produce **`build-ios/CyberCadKernel.xcframework`** with the headers from `include/`
  (`build-xcframework.sh:39-44`). The `.xcframework` is a build artifact and is gitignored; the
  app rebuilds it locally (`build-xcframework.sh:5-6`).
  - **Prerequisite.** The script needs OCCT iOS headers per slice at
    `$OCCT_ROOT/install-<tag>/include/opencascade` (default `OCCT_ROOT=/Users/leonardoaraujo/work/cybercad/build/occt`,
    `build-xcframework.sh:10,17`). This is the *transitional* xcframework (kernel + its own OCCT
    adapter); the terminal OCCT-free xcframework is produced once the M8 unlink is executed.
  - **Note (min iOS).** The script defaults to `MIN_IOS=13.0` while the app targets iOS 18.0
    (`project.yml:22`). Set `MIN_IOS=18.0` when building for parity with the app's deployment
    target (the script honours the `MIN_IOS` env var, `build-xcframework.sh:12`).
  - The macOS desktop equivalent is `scripts/build-macos-dylib.sh` (produces
    `build-mac/libcybercadkernel.dylib` via CMake `-DCYBERCAD_MACOS_OCCT=ON`) — relevant only if
    the app ships a Mac Catalyst / macOS target; the iPad app uses the xcframework.
- **Embed + link in the app (XcodeGen).** Because the project is XcodeGen-generated
  (`project.yml`), add the framework declaratively:
  - Add `CyberCadKernel.xcframework` under the target's `dependencies:` (or a `frameworks:`
    entry with `embed: false` — a **static** `.a`-backed xcframework is *linked*, not embedded,
    so no "Embed & Sign" copy phase is needed; there is no dynamic library to sign into the
    bundle). Regenerate the `.pbxproj` with `xcodegen generate`.
  - Point `HEADER_SEARCH_PATHS` at the xcframework's headers (they ship the `include/` tree, so
    the app includes `<cybercadkernel/cc_kernel.h>`).
  - **Signing.** A static xcframework contributes object code to the app binary, which is signed
    as part of the normal app-signing step — there is **no separate framework code-signing
    identity** to manage (unlike a dynamic `.framework`). This is the simplest signing story and
    is the recommended artifact for the app.
- **Retire the OCCT vendor once the kernel is OCCT-free.** Until the terminal unlink, the app
  keeps `Vendor/occt` linked (see §3(c) safety net). After the unlink, remove the
  `project.yml:34-41` managed OCCT block (the 22 `-lTK*` flags, the two search paths, the two
  `CYBERCAD_HAS_OCCT` defines) and delete `Vendor/occt` — the ~112 MB on-disk dependency the app
  stops vendoring/shipping/signing (BENCH-native-vs-occt.md §2).

### (b) Swap `KernelBridge.mm`'s OCCT implementation for the kernel ABI — the ABI/behavior deltas

The Swift side (`BRepShape.swift` and its call-sites) does **not** change: it already talks
plain C. What changes is that the `cc_*` symbols now resolve into `libcybercadkernel.a` instead
of `KernelBridge.mm`. The practical work is to **stop compiling the OCCT arm of `KernelBridge.mm`**
and let the linker bind the app's `cc_*` calls to the kernel.

**This is *nearly* a drop-in, with one concrete, enumerated delta the app MUST resolve:**

> **DELTA — 6 app-called functions are in the app's bridge header but NOT in the kernel product
> header.** These are live dependencies (all called from Swift via `BRepShape`), so a naïve link
> swap **fails to link**. Grounded in `KernelBridgeAPI.h` vs `include/cybercadkernel/cc_kernel.h`
> and verified call-by-call in app source:

| app-only `cc_*` (in `KernelBridgeAPI.h`, NOT in `cc_kernel.h`) | app wrapper | live call-site evidence |
|---|---|---|
| `cc_loft_circles` (`KernelBridgeAPI.h:63`) | `BRepShape.loftCircles` (`BRepShape.swift:226`) | called from `Workspace.swift` |
| `cc_loft_circle_wire` (`KernelBridgeAPI.h:69`) | `BRepShape.loftCircleToWire` (`BRepShape.swift:261`) | 2 sites in `Workspace.swift` |
| `cc_loft_along_rails` — **two-rail** (`KernelBridgeAPI.h:103`) | `BRepShape.loftAlongRails` (`BRepShape.swift:356`) | `RootView.swift`, `Workspace.swift` |
| `cc_loft_typed` (`KernelBridgeAPI.h:184`) | `BRepShape.loftTyped` (`BRepShape.swift:246`) | `Workspace.swift` |
| `cc_shape_solid_count` (`KernelBridgeAPI.h:350`) | `BRepShape.solidCount` (`BRepShape.swift:552`) | `Workspace.separateDisconnectedSolids` |
| `cc_shape_solid_at` (`KernelBridgeAPI.h:354`) | `BRepShape.connectedSolids` (`BRepShape.swift:559`) | `Workspace.swift` |

**Resolution (choose per function; all are app-integration work, no new native research):**
1. **Add the missing signatures to the kernel ABI** (additive-only; the ABI contract explicitly
   permits additive growth). The kernel already has the *general* forms the app-only variants are
   convenience wrappers around: `cc_solid_loft_wires` / `cc_solid_loft_sections` cover ruled
   lofting (`cc_kernel.h:313-327`), and the kernel can expose `cc_loft_circles` /
   `cc_loft_circle_wire` / `cc_loft_along_rails` / `cc_loft_typed` /
   `cc_shape_solid_count` / `cc_shape_solid_at` as thin additive entry points. **This is the
   recommended path** — it preserves the app's Swift layer verbatim. *(Kernel-product work,
   outside this doc's read-only scope, but small and additive.)*
2. **Re-implement the 6 in a slim app-side shim** in terms of the kernel primitives the app *does*
   get (e.g. `loftCircles`/`loftCircleToWire` as typed-profile lofts via `cc_solid_loft_wires`;
   `solidCount`/`solid_at` via `cc_subshape_ids`/`cc_boolean` decomposition). More app code, no
   kernel change.

Aside from those 6, **the shared surface is ABI-identical** (same signatures, same POD layouts,
same `0 = failure` convention). Everything the app calls in the FEATURE/BOOLEAN/QUERY/TRANSFORM/
EXCHANGE groups (`cc_fillet_edges`, `cc_boolean`, `cc_shell`, `cc_offset_face`, `cc_tessellate`,
`cc_subshape_ids`, `cc_mass_properties`, `cc_step_export`/`import`, all transforms, all reference
geometry) maps 1:1 to `cc_kernel.h`.

**Kernel functions the app does NOT use** (so no adaptation needed — they are additive kernel
capability): `cc_set_engine`, `cc_active_engine`, `cc_set_parallel`, `cc_set_gpu_tessellation`,
`cc_guided_orient_sweep`, `cc_chamfer_edges_asym`, `cc_project_point_on_face`, `cc_check_solid`,
`cc_stl_export`/`cc_stl_import`, `cc_step_pmi_scan`, `cc_hlr_project`, `cc_section_plane`,
`cc_tet_mesh`/`cc_mesh_quality`, `cc_fillet_edges_g2`, `cc_thread_apply`,
`cc_full_round_fillet[_faces]` — **0 app call-sites each** (verified in app source). Notably the
two *hard-decline* Class-B ops (`cc_fillet_edges_g2`, `cc_thread_apply`) have 0 app usage, so no
B-class op blocks the app.

**No behavior delta on the shared surface.** The `0 = invalid` return convention, the
`cc_last_error()` semantics, and the POD ownership rules (`cc_*_free`) are identical in both
headers. The app's Swift `wrap(id)` (`BRepShape.swift:18-20`, `id != 0 ? … : nil`) already
encodes the shared contract exactly.

### (c) Engine selection during the transition

The app currently *cannot* select an engine (§1). After adoption the app should drive the
selector the kernel provides (`cc_kernel.h:289-290`):

- **`cc_set_engine(0)` (DEFAULT).** Restores the build's default engine — OCCT where linked. This
  is the **safety net**: keep `Vendor/occt`-free-but-kernel-OCCT-linked so a native decline can
  still fall through to the kernel's own OCCT oracle, and so a global flip back to OCCT is one
  call away. The default is OCCT until `cc_set_engine(1)` is called explicitly (`cc_kernel.h:286`),
  so **existing behaviour is unchanged the moment the kernel is linked** — adoption alone changes
  nothing observable.
- **`cc_set_engine(1)` (NativeEngine).** Opts the process into native. Under it, native
  construct/tessellate/mass/bbox/etc. serve native bodies; out-of-envelope input self-verifies and
  falls through to OCCT (while linked). Shape ids created under one engine should be consumed
  under the same engine (`cc_kernel.h:288`); a native body is recognised process-wide, so
  toggling between build and boolean is safe (`cc_kernel.h:470-477`).
- **Staged rollout (see §4).** Rather than a global flip, the app can wrap individual operations
  in a `cc_set_engine(1) … cc_set_engine(0)` bracket (per-op opt-in) or run a shadow A/B, so the
  flip is reversible per operation. The app must add a small selection call-site (it has none
  today); the simplest is a single feature-flag guarded call at app start.

### (d) The decline-handling contract — confirm + the gaps

The kernel's clean-decline contract is *"return `id=0` with a non-empty `cc_last_error`, never a
fabricated handle"* (DROP-OCCT-READINESS.md §6.4). **The app is already resilient to this by
construction** — confirmed in source:

- **Universal `id==0 → nil`.** `BRepShape.wrap` (`BRepShape.swift:18-20`) turns any `CCShapeId==0`
  into `nil`. Every failable kernel op flows through it.
- **Edge edits surface the decline.** `Workspace.applyEdgeEdit` (`Workspace.swift:3464-3480`)
  guards on the `nil` result and, on decline, calls a user-facing
  `warn("<label> failed — the value may be too large for this geometry. Try a smaller one." +
  kernelDetail())` (`Workspace.swift:3471`), where `kernelDetail()` (`Workspace.swift:3483-3485`)
  appends the exact `cc_last_error()` as `" (kernel: …)"`. It **pushes no history and mutates no
  state** on failure — `pushBodyHistory` / `brepBody = result` run only on success. This is the
  fillet/chamfer/shell/offset path.
- **No crash path.** No `fatalError`/`assert`/force-unwrap of a kernel result exists in
  `BRepShape.swift` or `Workspace.swift`; a decline propagates as `nil → false` and returns early.
  This matches the M8 rehearsal's measured **0 crashes / 0 silent-wrong** (DROP-OCCT-READINESS.md
  §6.3).

**Two resilience gaps the app team should close (small, app-side):**

1. **IGES declines are SILENT (the named gap).** `Workspace.importIGES` (`Workspace.swift:2642-2650`)
   returns `false` on a `nil` import with **no `warn()`** (`Workspace.swift:2644`); `Workspace.exportIGES`
   (`Workspace.swift:2632-2637`) returns `nil` on a failed export with **no `warn()`**
   (`Workspace.swift:2637`) — the Share sheet simply never appears. Under native-only these
   become clean declines that fail silently. **Add a `warn()` on both paths** so the decline is
   honest to the user.
2. **Boolean decline is silent at `booleanWithBody`.** `Workspace.booleanWithBody`
   (`Workspace.swift:1046-1057`) returns `false` cleanly on a declined boolean (guard at
   `Workspace.swift:1049`) with **no `warn()` at that level** — state is left intact (safe), but
   the user gets no message unless a caller surfaces it. Confirm the combine call-site surfaces
   the `false`, and add a `warn()` if it does not. (The prior audit noted this as "returns false
   cleanly"; the subtlety is the absence of a user-facing message, same class as the IGES gap.)

### (e) The IGES product decision (decision memo)

IGES is the **only clean Class-C op the app uses** — under native-only both
`cc_iges_import`/`cc_iges_export` become clean declines (`native_engine.cpp:2062,2065`;
measured clean-decline in DROP-OCCT-READINESS.md §6.4). The app wires IGES end-to-end across
**7 live sites** (the prior audit's "11" counted the `KernelBridge.mm` bodies + `KernelBridgeAPI.h`
declarations too; the *live Swift/UI/test* sites are 7):

| # | site | role |
|---|---|---|
| 1 | `BRepShape.importIGES` (`BRepShape.swift:430-431`) | `wrap(cc_iges_import(path))` |
| 2 | `BRepShape.exportIGES` (`BRepShape.swift:664-665`) | `cc_iges_export(id,path) != 0` |
| 3 | `Workspace.importIGES` (`Workspace.swift:2642-2650`) | scale + commit imported body |
| 4 | `Workspace.exportIGES` (`Workspace.swift:2632-2637`) | scale + write export file |
| 5 | `ModelWorkspaceChrome.swift:71` | UI: open `.iges`/`.igs` → `workspace.importIGES(url)` |
| 6 | `ModelWorkspaceChrome.swift:195` | UI: `Button("IGES (B-rep)")` export |
| 7 | `BRepShapeIntegrationTests.swift:130-145` | round-trip export→reimport test |

**✅ DECISION MADE (2026-07-10): option (a) — DROP IGES from the app.** IGES is **not needed for
the foreseeable**; the app removes the two UI sites + the import/export/test paths, STEP remains the
exact-B-rep exchange, and `cc_iges_*` becomes a clean decline at unlink (no OCCT shim, no native
reimpl). **This clears the app's last Class-C blocker → a fully-OCCT-free app is reachable.** Native
IGES (option (c), ~1.5–3 py) is **parked on the backlog** — revisitable if IGES demand ever
materialises, but not scheduled. The three options are retained below for the record.

**Three options (for the record; (a) chosen):**

| option | what it means | cost / impact |
|---|---|---|
| **(a) Drop IGES** | Remove the two UI sites (`ModelWorkspaceChrome.swift:71,195`); leave STEP as the exact-B-rep exchange (`cc_step_export`/`import`, Class-A/native, @12/@8 app sites). | **Cheapest.** ~1 app dev-day: delete the import extension + export button + the round-trip test, add a migration note. STEP already covers exact B-rep exchange, so functional loss is limited to interop with IGES-only tools. **Recommended for the first OCCT-free ship.** |
| **(b) OCCT-linked IGES shim** | Keep a thin OCCT `IGESControl_*` shim (the existing `KernelBridge.mm:48-49` code) *outside* the native core, linked only for IGES. | **Medium.** Retains IGES functionality, but the **app is then not 100 % OCCT-free** — it still vendors the IGES-related toolkits (`-lTKDEIGES -lTKXSBase -lTKDE …`) and their transitive deps, undercutting most of the ~28 MB in-binary / ~112 MB dependency size win. Only worth it if IGES interop is a hard customer requirement. |
| **(c) Native IGES reimpl** | Implement IGES import/export natively in the kernel (like the native STEP reader). | **Highest — ~1.5–3 py, out of current scope** (DROP-OCCT-READINESS.md §3). Delivers full IGES with zero OCCT. Defer unless a market need justifies it. |

**Recommendation:** ship with **(a)** (drop IGES; STEP covers exact exchange), keep **(c)** on the
backlog if IGES demand materialises. **(b)** only if a customer contract mandates IGES *and* the
size regression is acceptable. Whichever is chosen, first close the silent-decline gap in §3(d)(1)
so an IGES failure is never invisible.

## 4. Risk / validation — how the app team validates parity, and a reversible rollout

**Parity oracles available to the app team:**

- **The kernel's own native-vs-OCCT sim suite.** Every kernel capability is gated by the
  non-negotiable two-gate discipline (MOAT-ROADMAP.md): (a) a host-analytic closed-form check,
  and (b) a sim native-vs-OCCT parity check on a booted iOS simulator comparing volume / area /
  watertightness / topology against the OCCT oracle. The `scripts/run-sim-*` suite (60+ runners,
  e.g. `run-sim-native-boolean.sh`, `run-sim-native-curved-fillet.sh`, `run-sim-suite.sh`) is the
  differential harness. The app team should run `run-sim-suite.sh` on the exact xcframework build
  it links, so parity is validated on the shipped artifact.
- **The app's own ~170 tests.** Run the full app test suite under `cc_set_engine(0)` (OCCT
  baseline) and again under `cc_set_engine(1)` (native), diffing results. The M8 rehearsal already
  measured the equivalent on the kernel's host suite: **53 PASS-native / 0 crash / 0 silent-wrong**
  (DROP-OCCT-READINESS.md §6.3), with the only "failures" being 3 sentinel tests that assert the
  *stub-default* invariant the rehearsal deliberately inverts (not regressions).
- **The M6 completeness/fuzz bar** is the true gate before the terminal OCCT unlink: differential
  fuzzing across 13 native domains, **0 DISAGREED**, guaranteeing no silent-wrong once the oracle
  is gone (DROP-OCCT-READINESS.md §6.7). The app should not remove `Vendor/occt` until this holds.

**Staged, reversible rollout:**

1. **Adopt with default OCCT.** Link the kernel, `cc_set_engine(0)`. Observable behaviour is
   identical to today (the kernel's OCCT adapter serves everything). This de-risks the *link swap*
   independently of the *engine flip*.
2. **Per-op native opt-in.** Bracket individual ops (`cc_set_engine(1)` around the call,
   `cc_set_engine(0)` after) — start with the high-confidence, high-volume A-spine ops
   (extrude/revolve/tessellate/mass/subshape/transforms) that the rehearsal proved SERVED-NATIVE.
   Any decline degrades to the app's existing `warn()` (§3(d)); nothing breaks.
3. **Shadow-compare (optional).** For a canary period, run both engines and compare (volume /
   bbox / mesh triangle count) before trusting native; log divergences. This is exactly the sim
   suite's methodology applied in-app.
4. **Global flip.** `cc_set_engine(1)` process-wide once the per-op and shadow phases are clean.
   Reversible with a single call.
5. **Terminal unlink.** Rebuild the kernel OCCT-free, drop `Vendor/occt` and the OCCT block in
   `project.yml:34-41`. Gate on M6 (step above).

**Payoff that justifies the flip** (measured, BENCH-*.md): native is **7–20× faster** on the hot
interactive path (boolean/tessellate/mass), **~28 MB lighter in-binary** (dead-stripped final
link) / **~112 MB lighter on-disk dependency**, and **~11.5 MB lower peak RSS** (35.3 MB vs 46.8 MB
on the same representative script) — the tightest iPad axis.

## 5. What adoption delivers today vs what still forwards to OCCT

**Delivered natively today (adopt + flip), by call-site weight the large majority of app geometry:**

- The **full Class-A spine**: construct (extrude/revolve/loft/sweep + holes/profiles),
  **all transforms** (M-TX: translate/rotate/mirror/scale/place_on_frame — 27 app sites),
  **all reference geometry** (M-REF: face_axis/tangent_chain/outer_rim_chain/offset_face_boundary
  — 22 app sites), **all query/analysis** (subshape_ids @17, tessellate @12, face_meshes @9,
  edge_polylines @8, mass_properties @7, principal_moments @6, bounding_box @4), and
  **STEP import/export** (@8/@12). The M8 rehearsal MEASURED every A-spine op SERVED-NATIVE
  (DROP-OCCT-READINESS.md §6.4).
- **In-envelope blends/booleans**: the typical prismatic case of `cc_fillet_edges` /
  `cc_chamfer_edges` / `cc_shell` / `cc_offset_face`, and planar + analytic-curved `cc_boolean`
  (the bulk of CAD booleans) — all native.
- **Closed by THIS session's M3/M2 work** (narrower gap than the prior audit implied):
  - **Curved-edge fillet on analytic-revolve substrates** — `cc_fillet_edges` (the app's
    highest-usage blend, @15) is now **native on cylinder + cone-frustum + sphere cap-rims**, i.e.
    the revolved/tapered/truncated-ball rims the app actually exercises (DROP-OCCT-READINESS.md §1
    row 1204, §2 M3). The prior audit's "curved-edge fillet is the single largest gap" is now
    *largely closed* for the app's analytic-revolve cases.
  - **`cc_fillet_face`** — native on a planar prism cap via the spherical corner-weld
    (DROP-OCCT-READINESS.md §1 row 1440); @7 app sites covered in-envelope.
  - **Curved shell** — capped-cylinder / cone-frustum `cc_shell` now native (M3).
  - **The M2 freeform-boolean spine** (half-space → corner-clip → curved-wall CUT+COMMON → ff↔ff
    → ≥3-seam) is complete and welds vs OCCT — the app's `cc_boolean` @13 is served for far more
    than the planar case.

**Still forwards to OCCT after adoption (the A-envelope residual the app should expect to
decline under native-only) — smaller than before, and none is a hard-decline B op:**

| still-forwarding op | app sites | status this session |
|---|---:|---|
| `cc_fillet_edges` on **non-analytic** curved crease (cyl↔cyl canal, torus/elliptical crease, freeform edge) | subset of @15 | **PENDING M3** — needs a new tessellator crease weld / freeform substrate; the analytic-revolve rims are already native (above) |
| `cc_offset_face` on **curved** faces | subset of @10 | **PENDING M3** curved bucket |
| `cc_shell` on **stepped / sphere / freeform / non-convex** bodies | subset of @12 | **PENDING M3** (capped-cyl/cone landed) |
| `cc_boolean` **freeform / all-OCCT-operand** residual | subset of @13 | **PENDING M2 breadth** (spine landed) |
| `cc_wrap_emboss` on a **freeform base** | subset of @6 | **PENDING M3 freeform-base** (cylinder base native) |
| `cc_helical_thread` / `cc_tapered_thread` **fine-pitch self-intersecting** | subset of @6/@5 | **PENDING M7b** (coarse pitch native) |
| `cc_iges_import` / `cc_iges_export` | 7 live sites | **DROPPED (Class C)** — product decision §3(e), not a kernel gap |

Under native-only each of these is a **clean decline** (id=0 + non-empty `cc_last_error`), which
the app already degrades to an honest `warn()` (§3(d)) — **not a crash, not a corrupted body**
(measured: DROP-OCCT-READINESS.md §6.3–6.5). The remaining kernel work is bounded breadth
(~2.5–5 py, midpoint ~3.5 py; DROP-OCCT-READINESS.md §2) on machinery that already works for the
common case — and it is the **same M2/M3/M7b moat the roadmap already tracks**; adoption creates
no new kernel work.

**Bottom line for the app team:** *Adopt now* delivers a native app that runs the overwhelming
majority of real geometry natively (faster + lighter), degrades every gap to an honest decline,
and is reversible per-op. The finite work to make it *feature-complete* vs the OCCT build is
(1) the 6 app-only ABI shims in §3(b), (2) the IGES decision in §3(e), (3) the two silent-decline
UX fixes in §3(d), and (4) the kernel-side M2/M3/M7b curved-blend/freeform residuals in §5 — none
of which is new, and none of which involves a hard-decline B op the app calls.
